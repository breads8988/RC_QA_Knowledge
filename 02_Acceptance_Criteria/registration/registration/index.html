    User Story¶

    As a new (unauthenticated) Repaircheck web-app user I want to create an account through a guided multi-step flow (personal info → email verification → address → password → vehicle) So that I can register my vehicle(s) and use the app's repair features.

    Source: RC-8 — Web app – Implement UI of the Registration page (child of Epic RC-21 User management). Grounded against the Figma screenshots in 01_SRS/registration/figma.

    Business Rules (rule-based AC)¶

    Rule ID Business Rule Jira Rationale / Source Criticality Linked TCs Status
    BR-REG-01 An email address may be registered only once — a second registration attempt with an existing email is rejected. RC-8 RC-8 ("Email has already been registered") 🔴 Critical TC-REG-005 Draft
    BR-REG-02 The email must be a syntactically valid email address before Step 1 can be submitted. RC-8 RC-8 ("Invalid email format") 🟠 High TC-REG-004 Draft
    BR-REG-03 Password must meet ALL of: ≥8 characters; ≥1 symbol or number; ≥1 capital letter; contains no spaces. The continue action is enabled only when all four pass. RC-8 RC-8 Step 3 password rules 🔴 Critical TC-REG-015, TC-REG-016, TC-REG-017 Draft
    BR-REG-04 Email ownership is verified by a 6-digit code sent to the entered address; the correct code must be entered to proceed past verification. RC-8 RC-8 Step 1 verification 🔴 Critical TC-REG-003, TC-REG-008 Draft
    BR-REG-05 The verification code can be resent only after the resend cooldown expires (5 minutes). RC-8 Stakeholder (5 min) — ⚠️ Figma shows "00:23" 🟡 Medium TC-REG-010 Draft
    BR-REG-06 Required fields — Step 1: email, first name, last name; Step 2: postcode, street name & number. Each step cannot be completed until its required fields are filled. RC-8 RC-8 Steps 1–2 🟠 High TC-REG-006, TC-REG-011, TC-REG-014 Draft
    BR-REG-07 The postcode must be in a valid format before Step 2 can be submitted. RC-8 RC-8 ("Please enter a valid postcode") 🟡 Medium TC-REG-012 Draft
    BR-REG-08 At least one vehicle must be registered to complete the registration flow. RC-8 RC-8 Step 4 🟠 High TC-REG-022 Draft
    BR-REG-09 A user may register a maximum of 4 vehicles; beyond that, adding is blocked and a contact-support popup is shown (Alexis@repaircheck.com). RC-8 RC-8 Step 4 / RC-20 (my_vehicle) 🟡 Medium TC-REG-025, TC-REG-026 Draft
    BR-REG-10 A verification code is valid for 5 minutes; once it expires it is rejected and a new code must be requested. RC-8 RC-8 Step 1 verification (5-min expiry, confirmed) 🟡 Medium TC-REG-030 Draft
    BR-REG-11 Vehicle ownership is a mandatory single choice between "Owner" and "Rental", pre-selected to "Owner" by default; exactly one is always selected. RC-8 Stakeholder decision (2026-07-04) — default Owner 🟡 Medium TC-REG-038, TC-REG-021 Draft

    Open Questions / Assumptions (BA — confirm with stakeholders before TC generation)¶

    1. Step 4 scope (vehicle registration) — DECIDED: in scope. RC-8's comment "Skip Register Vehicle step for now" is disregarded per stakeholder decision (2026-07-01); the Register Vehicle step is covered (AC-REG-17, AC-REG-20…26, BR-REG-08/09). Note the vehicle detail overlaps RC-20 (my_vehicle feature, code MV) — keep the two registers in sync to avoid divergence.
    2. Vehicle field value ranges. The vehicle-form option lists (Brand/Modell/Baujahr/Kategorie/Motor come from API; Baujahr 1990–now; No. of doors 2–6; No. of seats 2/3/4/5/7; KW free text) are specified in RC-20, not RC-8. Confirm these ranges are authoritative for the registration flow too.
    3. Verification code details. Confirmed: 6-digit code, expiry = 5 minutes (drives AC-REG-28 / BR-REG-10), resend cooldown = 5 minutes (BR-REG-05), no max-attempt lockout (unlimited incorrect entries — no lockout case designed). ⚠️ Discrepancy to resolve with design: the Figma verification screen shows "Send again in 00:23" (23s), which conflicts with the confirmed 5-minute cooldown — align the mockup or the value.
    4. Password bounds. RC-8 gives a lower bound (≥8) and composition rules but no maximum length. Confirm max length and whether a strength label other than Weak/Strong exists (e.g. Medium).
    5. Name validation. The digits-only name produces a soft warning ("Are you sure…") in Figma — assumed non-blocking (AC-REG-07). Confirm it never blocks submission and whether any hard character rules apply (min/max length, allowed characters).
    6. Postcode validity. Assumed a format check only (Germany-oriented, per the German field labels). Confirm whether the postcode is validated against a real postcode service / must match the map-selected address.
    7. Email change before verification. If the user goes back and edits the email after a code was sent, is a new code required? (Not specified.)
    8. Deleted-account email reuse (AC-REG-27). Assumed an email freed by account deletion becomes available for a fresh registration. Confirm whether deleted-account emails are truly released, reserved for a grace period, or permanently blocked.
    9. Vehicle ownership field — DECIDED (2026-07-04). The "Owner / Rental" choice belongs to registration (this feature), not MV. It is a required single choice defaulting to "Owner" (pre-checked); the user can switch to "Rental". Covered by AC-REG-35 / BR-REG-11. The same field on the MV vehicle form is documented here and intentionally not duplicated in my_vehicle.
    10. Password visibility (AC-REG-30). Assumed the field is masked by default with an eye icon that reveals and re-hides. Confirm the default state and whether the toggle persists across steps.
    11. SRS note. No 01_SRS/registration/epic.md exists yet; SRS ref points at the folder. Create the SRS note if the team wants a formal SRS anchor.
